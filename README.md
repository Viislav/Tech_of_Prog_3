# ОТЧЁТ

Ветка main отвечает за проверку общей работоспособности, где нет никаих ошибок. В то же время 2 ветка отвечает за показ работоспособности проверки в условиях ошибки(которую мы специально вывели).

Ключевой задачей(и самой главной проблемой) было настроить Trivy. Его диагностику я провожу не через sarif, а через саму диагностику в GitHub.
В конечном счете я его настроил следующим образом:

	Trivy FS - Задание сканирует только файл requirements.txt на наличие уязвимостей (благодаря scan-type: 'fs' и scanners: 'sbom,vuln'). Старая версия requests (2.27.1) имеет уязвимости, но они часто связаны с транзитивными зависимостями (urllib3, certifi) или являются проблемами, которые Trivy не может обнаружить без полной установки (то есть без pip install -r requirements.txt).
	Trivy Image - Задание сканирует готовый Docker-образ. Образ содержит все установленные зависимости, включая уязвимую версию requests и все ее транзитивные уязвимые подзависимости (urllib3, certifi). Эти уязвимости не являются пакетами ОС, поэтому фильтр Rego их не игнорирует.
	Всё остальное в ci - настроено как было показано в примерах, ведь оно сразу заработало корректно.
